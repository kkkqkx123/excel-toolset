# 一、先明确：Excel SQL 查询是什么

常规 Excel 工作表可以被**虚拟成关系型数据库的数据表**：

- 工作表 = 数据表
- 行 = 数据记录
- 列 = 字段/列名

**Excel SQL 查询**，就是使用标准 SQL 语法（`SELECT`/`WHERE`/`GROUP BY`/`JOIN`/`LIMIT` 等），对 Excel 内的数据执行**查询、筛选、聚合、关联、计算**。

结合你的**无头 Rust 环境**：
本项目并不会依赖 Excel 桌面端的 Power Query/Microsoft Query，而是采用**内嵌轻量 SQL 引擎**（DuckDB / Apache DataFusion / SQLite）实现：

1. 通过 `calamine` 读取 Excel 数据到内存；
2. 把数据载入内存虚拟数据表；
3. 接收外部传入的 SQL 语句并执行；
4. 将结果转为结构化 JSON 返回给 CLI/HTTP 调用方（AI Agent）。

> 核心特性：**纯只读操作**（默认不修改原 Excel 文件），属于「数据查询分析」能力，而非单元格编辑能力。

# 二、Excel SQL 查询的核心作用

## （一）通用数据处理作用（传统自动化场景）

对应你之前划分的「高级操作」，用来弥补**普通单元格操作、筛选、排序接口**的能力短板：

### 1. 实现复杂多条件筛选

普通原子接口（`excel_filter`）一般只支持简单条件（等于、大于、包含），多条件组合、嵌套条件实现繁琐。
SQL 可直接用 `AND/OR/IN/LIKE/()` 组合复杂规则，例如：

> 筛选「部门=市场 **且** 业绩>10000 **或** 入职年份<2023」的数据
> 一条 SQL 即可完成，无需多次调用筛选接口叠加逻辑。

### 2. 按需投影列，减少数据传输与内存占用

日常读取区域需要指定 `A1:Cxx` 这类单元格范围，容易读取冗余列。
通过 `SELECT 列1,列3` 可以**只查询需要的字段**，尤其面对几十列的大表格时，大幅减少 IO 和数据体量。

### 3. 分组聚合、批量统计（替代手工公式）

这是 SQL 最核心的价值之一，原生 Excel 需要手动写 `SUM/COUNT/AVG` 公式、设置数据透视表；
SQL 用 `GROUP BY + 聚合函数` 一步完成分组统计：

- 按部门分组统计人数、平均业绩；
- 按日期汇总每日数据；
  无需在 Excel 中新增公式单元格、透视表，直接产出统计结果。

### 4. 多工作表关联查询（多表 JOIN）

Excel 原生很难实现跨 Sheet 联动查询，而 SQL 支持 `JOIN`，可以把**多个工作表当作多张数据表**做关联匹配，例如：

- 员工表 + 工资表 联查，匹配员工姓名与对应薪资。

### 5. 一站式排序、去重、分页

整合 `ORDER BY`（排序）、`DISTINCT`（去重）、`LIMIT/OFFSET`（分页）：
大表格分页读取、自动去重、多列复合排序，**一条 SQL 替代多次调用排序/去重接口**。

### 6. 字段级衍生计算与分支逻辑

支持列运算、`CASE WHEN` 条件判断，实现复杂数据加工：

- 两列数值相除得到比率；
- 根据分数自动划分「优秀/合格/不合格」等级；
  无需在 Excel 中新建辅助列、写入临时公式。

## （二）面向 AI Agent 的专属核心价值（重点，贴合你的项目）

这也是你这套工具**必须保留 SQL 查询能力**的关键原因，区别于普通 Excel 自动化：

### 1. 大幅减少工具调用轮次，降低 LLM 幻觉

如果不用 SQL：
复杂分析任务需要 LLM 依次拆解、串行调用多个原子接口：
`读取区域 → 筛选 → 排序 → 分组统计`
每一次调用都是一轮对话交互，**轮次越多，LLM 越容易出错、产生幻觉**。

使用 SQL：
LLM 把一整条分析需求翻译成一句 SQL，**仅一轮调用**即可拿到最终结果，链路更短、稳定性更高。

### 2. 适配 LLM「自然语言转 SQL」的原生能力

大模型天生擅长将人类自然语言描述，翻译成标准 SQL 语句，这是目前 Agent 数据处理的主流范式：

- 人类指令：`统计销售部所有业绩大于5000的员工总数`
- LLM 自动翻译为 SQL → 调用本项目 `excel_sql_query` 接口 → 返回结果

相比让 LLM 记忆 Excel 单元格坐标、区域范围、行号列号等**Excel 专属语法**，SQL 语法更通用、LLM 理解成本更低。

### 3. 统一查询范式，降低工具记忆成本

你的工具已有十几类原子操作（读写单元格、行操作、样式、VBA 等），LLM 需要记忆大量不同命令/参数规则。
SQL 提供**一套通用语法**：单表、多表、筛选、统计全部用 SQL 表达，LLM 不需要区分各类零散 Excel 操作接口。

### 4. 大文件场景性能更优

底层基于 DuckDB/DataFusion 这类列式查询引擎，针对海量行数据做了查询优化，比逐行遍历 Excel 单元格的普通读取接口更快，适合 Agent 处理数万行级别的报表。

### 5. 输出格式天然适配后续链路

SQL 查询结果是**标准行列结构化数据**，和你项目统一的 JSON 响应、Diff 比对、数据回填逻辑完全兼容，拿到结果后可直接用于二次加工、写入 Excel、差异对比。

# 三、明确边界：什么时候用 SQL，什么时候用原有原子接口

结合功能定位，划分使用场景，方便后续 Agent 策略配置：

## ✅ 优先使用 Excel SQL 查询（分析/统计/复杂筛选）

1. 多条件组合筛选、模糊匹配、区间查询；
2. 分组统计、求和/计数/平均值等聚合计算；
3. 多个工作表跨表关联查询；
4. 大表格分页读取、抽取指定列；
5. 自然语言驱动的数据分析、报表统计类需求。

## ❌ 不建议使用 SQL（走原有原子接口）

1. **单元格/行/列编辑**：单格写入、区域修改、插入/删除行、合并单元格、设置样式/字体/边框（SQL 只查不改，不处理格式与布局）；
2. **文件/工作表管理**：新建文件、增删工作表、备份、VBA 读写；
3. **简单操作**：仅读取少量单元格、单条件简单筛选（用 `excel_read_range`/`excel_filter` 更轻量，无 SQL 引擎开销）；
4. **公式设置**：向单元格写入 Excel 公式（SQL 不负责单元格公式写入）。

# 四、结合你现有架构的集成设计（无重构，平滑嵌入）

基于之前敲定的 **两层扁平架构** + 模块划分，SQL 查询能力归属、执行流程、规则如下：

## 1. 模块归属

- 所属模块：`excel_data.rs`（数据处理模块，和筛选、排序、去重放在一起）
- 对外原子工具名：`excel_sql_query`（和你之前规划的工具清单一致）
- 新增依赖：内嵌 SQL 引擎（推荐 `duckdb`，Rust 生态成熟、轻量、嵌入式部署，无独立服务）

## 2. 完整执行链路（纯查询，无文件修改）

```
CLI/HTTP 请求 → 入口层（解析文件路径、Sheet 名、SQL 语句）
→ excel_data::excel_sql_query
   1. 调用 excel_read + calamine 读取 Excel 数据
   2. 将数据载入 DuckDB 内存虚拟表
   3. 执行传入的 SQL 语句
   4. 结果转为结构化二维数组
→ security 模块：计算文件哈希（file_hash，统一响应规范）
→ 组装标准 JSON 响应（success + message + data + file_hash）
→ 返回调用方
```

### 关键规则（适配现有安全/规范）

1. **纯只读**：不修改原 Excel 文件，**不触发备份、不启用 dry_run、不生成 Diff**；
2. 仅复用全局「文件指纹」能力，其余写操作相关逻辑全部跳过；
3. 安全加固：增加 **SQL 参数化**，禁止直接拼接用户输入，防止 SQL 注入（Agent 场景必备）；
4. 支持多 Sheet：允许将多个工作表注册为多张内存表，支持 `JOIN` 跨表查询。

## 3. 入参/出参规范（沿用全局统一格式）

### 通用入参

- 必传：`file_path`（Excel 文件路径）、`sql`（SQL 语句）
- 可选：`sheet_mapping`（多表场景：工作表名 → 数据表别名）

### 通用出参

完全复用你项目全局响应结构：

```json
{
  "success": true,
  "message": "SQL执行成功",
  "file_hash": "xxx",
  "data": [
    ["行1列1", "行1列2"],
    ["行2列1", "行2列2"]
  ],
  "diff": null,
  "backup_info": null
}
```

## 4. 与公式引擎、Diff 能力的联动

1. SQL 查询**读取的是单元格原始值**（XML 中 `<v>` 缓存值）；如果需要拿到公式实时重算结果，先执行 `excel_refresh_calc` 重算，再执行 SQL 查询；
2. 若需要「把 SQL 查询结果写回 Excel」：分两步走
   - 第一步：调用 `excel_sql_query` 查询数据；
   - 第二步：调用 `excel_write_range` 写入区域，此时才会触发备份、Diff、dry_run 等写操作逻辑。

# 五、总结

1. **本质**：Excel SQL 查询 = 把表格数据库化，用标准 SQL 语法做**数据查询、筛选、统计、多表关联**，属于**只读分析能力**。
2. **核心价值**
   - 通用场景：简化复杂筛选、分组统计、跨表查询，替代零散手工操作；
   - AI Agent 场景：**减少调用轮次、降低 LLM 幻觉、适配自然语言转 SQL 能力**，是数据类 Agent 的核心工具。
3. **架构定位**：嵌入现有 `excel_data.rs` 模块，纯只读链路，不改动原有读写、安全、Diff 逻辑，集成成本极低。
4. **使用原则**：复杂分析用 SQL，简单读写/格式操作用原有原子接口，两者互补。
